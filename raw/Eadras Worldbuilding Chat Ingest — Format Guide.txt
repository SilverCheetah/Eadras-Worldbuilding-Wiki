Eadras Worldbuilding Chat Ingest — Format Guide
Purpose: Reconstructed format spec for single-doc Grok/ChatGPT chat ingest files in the Eadras worldbuilding project, based on conventions established across ~23 prior docs and refined through the 5-10/5-11/5-12 ingest session.
Status: Reconstructed from handoff description and in-session usage. Should be checked against an actual anchor file from playground_session/grok_5-8/ once located.

1. PURPOSE OF AN INGEST DOC
An ingest doc takes one source chat (Grok, ChatGPT, etc.) on one topic and converts it into a standalone canonical document that:

Captures the new canon introduced in that chat — locked claims about the world.
Cross-references the authoritative wiki to distinguish what's genuinely new vs. what the wiki already covers.
Flags contradictions — between the chat and the wiki, between different parts of the chat, or internal to the chat itself.
Preserves author craft notes — the meta-level reasoning behind in-world choices.
Enumerates open prompts — unresolved questions worth carrying into future sessions.

One source chat may produce one doc or several, depending on whether the chat covers one topic or jumps between several. Default is one topic per doc, even if the source chat is messy.
2. HEADER BLOCK
Every doc opens with a header:
# [Title — descriptive, not the chat's filename]

**World**: Eadras
**Source**: [Chat filename(s), date(s)]
**Document type**: Single-doc Grok ingest — [new canon / canon supersession / diagnostic / etc.]
**Companion documents**:
- [Other ingest docs that handle related material from the same or adjacent sessions]
- [Wiki articles that this doc augments or contradicts]

**Last updated**: [YYYY-MM-DD]

---
The title should describe the content, not the chat (e.g., "Dragon Empire — Far-Eastern Meritocracy" not "Grok 5-12 Ingest").
3. "HOW TO READ THIS DOCUMENT" SECTION
A short opening section that orients a future reader. Typical contents:

What kind of doc this is (seed canon / supersession / diagnostic-only / craft notes).
Whether the material is already in the wiki and how this doc relates to it.
The key flags that appear in the doc and what they mean (if the doc uses any non-standard ones).
The structure of the rest of the document.

This section is conversational rather than formal. It's the entry-point for someone who hasn't seen the doc before.
4. SECTION LABELS
Every substantive section is tagged with one or more labels indicating what kind of content it carries:

[IN-WORLD] — The section describes Eadras as it exists. Facts, history, peoples, places, mechanisms.
[META] — Author craft notes. Why a choice was made, what inspirations were drawn on, how the material works as authorial structure rather than as in-world fact.
[DIAGNOSIS] — Tracking the chat's reasoning process. Discarded options, alternatives the chat considered and rejected, why the final answer was chosen. Preserved because the reasons are instructive even when the rejected options aren't canon.

A single section can carry multiple labels (e.g., [CANONIZED] [IN-WORLD] is the most common pairing).
5. CANON FLAGS
Inline flags marking the canonical status of each claim. These are the load-bearing convention of the project — they let a future reader see at a glance what's locked, what's wiki-canonical, what's contradicted, and what's open.
The standard flag set:

[CANONIZED] — This is locked canon. The chat established it; the author confirmed it; it's authoritative.
[CANONIZED_AS X] — The chat said one thing; the canonical form is X. Used when the chat used a different name or framing than the wiki's locked version (e.g., chat says "Ssar'veth" but the scholarly form is Kudur-eṭir).
[WIKI_EXPANDS] — The chat introduces material that the wiki already covers in more detail. The wiki is authoritative; the chat material is consistent with it but doesn't supersede it.
[WIKI_REFINED] — The chat refines a wiki claim with additional nuance, where the wiki version is correct but incomplete.
[SUPERSEDED_BY_WIKI] — The chat introduced something that the wiki has since revised away. The wiki is authoritative; the chat material is recorded for completeness but is not canon.
[CHAT_SUPERSEDES_WIKI] — Introduced this session, used sparingly. The chat establishes new canon that explicitly overrides existing wiki canon. Default rule is that the wiki always supersedes the chat except where the author explicitly states otherwise. Used in the Spear doc for the new Spear-prison sequence.
[INTERNAL_CONTRADICTION] — The chat contradicts itself (e.g., tries one sequence, then another). The doc records both and flags the conflict.
[FLAG_FOR_REVIEW] — Open canon. The chat surfaced the question but didn't settle it; the author hasn't locked an answer yet.

Default rule: the wiki always supersedes the chat unless the author explicitly states otherwise. [CHAT_SUPERSEDES_WIKI] is the exception flag, not the norm.
6. STANDARD STRUCTURE
Most ingest docs follow this rough sequence after the header and "How to Read":

Core premise / load-bearing principle — the keystone claim of the doc.
Substantive sections — the actual canon material, one section per subtopic, with appropriate labels and flags.
Diagnosis section (if relevant) — discarded alternatives the chat considered.
Author craft notes — meta-level observations about why this material works.
Cross-references and integration hooks — how this doc connects to existing wiki articles and other ingest docs.
LIVE WIRES — closing enumeration of open prompts.

Not every doc needs every section. A purely diagnostic doc might have no in-world sections; a seed-canon doc might have no diagnosis section. The structure is a default, not a template to fill mechanically.
7. THE LIVE WIRES SECTION
Every doc ends with a LIVE WIRES section — a bulleted list of open story prompts, unresolved questions, and threads worth pulling in future sessions.
Live Wires entries are typically a sentence or two each, framed as a question or a prompt rather than a statement. They are not action items; they are creative invitations for the next session.
Example shape:


When did the dragon mercenary coup happen? Pinning the Empire's founding on the timeline determines its relationship to the Firethorn Empire and the Demon War.
What do dragons wear to display their ranks? Plaques scaled to body size? Magically projected pictograms? Inscribed scale-etchings?


8. SIZE TARGETS

Typical doc: 20-40KB.
Over 40KB: justified only when the content is genuinely load-bearing (recitation texts, full wiki-supersession sequences). The 5-10/5-11/5-12 session produced two docs in the 50KB range; both were judged justified.
Under 20KB: usually a sign the doc is thin or could be folded into a larger one.

9. TONE AND VOICE

Prose, not bullet lists for substantive canon claims. Bullets are reserved for Live Wires and for genuinely enumerable items (e.g., the four prison attempts).
Declarative for canon, careful for uncertainty. A [CANONIZED] claim is stated flatly. A [FLAG_FOR_REVIEW] claim is framed with appropriate hedges.
Quote the source chat sparingly but exactly when a specific phrase carries weight. The author's own framing for load-bearing principles should be preserved verbatim where possible.
The voice is institutional-archival — like a careful scholar writing for a future scholar. Not novelistic, not casual.

10. FILE NAMING AND LOCATION

Filename: descriptive-kebab-case.md. Describes the topic, not the source chat. (e.g., dragon-empire-far-eastern-meritocracy.md, not grok-5-12-doc-1.md.)
Location: /mnt/user-data/outputs/playground_session/[chat-set-name]/. The chat-set folder groups docs from a single ingest session (e.g., grok_5-10_11_12/).

11. WORKFLOW PER DOC
For each doc:

Read the source chat section on the topic.
Cross-reference the wiki for any related existing canon.
Decide on flag assignments: what's [CANONIZED], what's [WIKI_EXPANDS], what's [FLAG_FOR_REVIEW], etc.
Write the doc following the standard structure.
Check size; revise if over 40KB without justification.
Copy from working directory to outputs folder.
Call present_files to share.

12. KEY PRINCIPLES TO REMEMBER

One topic per doc. Don't merge unrelated material because it came from the same chat.
Wiki supersedes chat by default. Exceptions require explicit author direction.
Diagnosis preserves the reasoning, not just the conclusion. Discarded options matter.
Live Wires are creative prompts, not task lists. Frame them as invitations.
The reader is a future scholar. Write for someone reconstructing canon a year from now.
